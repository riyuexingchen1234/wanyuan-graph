# 万源图谱 项目说明文档

> 本文档用于向 AI 编程协作工具（如 Claude Code）对齐项目背景、核心设计决策与当前阶段目标。
> 文档记录的是经过多轮讨论收敛后的结论，请按此执行，不要重新发明已经讨论定下的结构。

---

## 1. 项目是什么

**创意名称**：万源图谱 —— 看见被行业分类切断的连接

**核心问题**：一种材料、一种工艺、一台设备，往往本可以服务多个互不相关的行业，但人们习惯按"行业"认识世界，导致这些真实存在的连接被切断、被忽视。从业者知道自己行业里的真相，外人却看不到；想找一个被市场忽略的细分机会的人，更是无从下手。

**项目想做的事**：把这些被行业语言分散开的信息，重新整理、用统一的客观语言描述，并排放在一起，让原本因为分类习惯而看不见的连接重新被看见。

**产品形态**：Web 端的产业关系图谱网站。用户可以从一个行业出发查看上下游关系，也可以从某个具体产品的核心材料/工艺出发，看到它延伸出的其他应用领域。

---

## 2. 项目的使命定位（重要，决定了所有后续设计原则）

这个项目被明确定位为**"图书馆使命"**，不是"实验室使命"：

- **图书馆使命**：整理已经存在但因为行业语言分散而被遮蔽的信息，让人重新并排看到。
- **实验室使命（不是本项目要做的）**：凭空发明、推导世界上还不存在的新知识。

这个定位决定了：图谱本体必须**客观、可溯源、不带价值判断**。任何"机会大小""门槛高低""该不该投资"之类的判断，都不属于图谱本体，应该被划到独立的分支产品中（见第6节）。

---

## 3. 核心数据结构：两套坐标系 + 交汇点

这是整个项目最核心的结构设计，所有数据建模和呈现逻辑都建立在这个结构之上。

### 坐标系 A：产业链关系网
- 节点性质：行业 / 产品类别
- 连接性质：供应链上下游关系（上游/下游）
- 用户从这里"进门"——大多数用户的思考起点是行业（光伏、储能这类概念），不是材料属性

### 坐标系 B：材料属性延伸网
- 节点性质：材料 / 工艺 / 设备的底层属性
- 连接性质：可被加工成 / 可应用于 / 结构相似（横向跳转）
- 这是项目真正的差异化价值所在——揭示同一种底层能力可以延伸到哪些目前未被关联的应用领域

### 交汇点（关键机制）
像"电池隔膜"这样的具体事物，**同时存在于两套坐标系**：它是坐标系A里"储能设备的上游"，同时也是坐标系B里"聚乙烯/聚丙烯的一种已知应用形态"。

**横向发现的真正触发机制**：不是靠算法主动推荐相似项，而是让每个交汇节点诚实地暴露它在两套坐标系里的位置，用户沿着熟悉的产业链路径走到交汇点时，自然"换轨"切入材料属性视角，由此看到原本想不到的延伸方向。

### 已验证 vs 待验证（图谱可信度的底线）
每一条连接（边）必须显式标注状态：
- **verified（已验证）**：有真实案例/来源支撑，必须可追溯（专利、行业报告、行业标准、官方数据等）
- **proposed（待验证）**：有依据支持的潜在可能性，但尚无真实案例验证，必须注明提出方式和理由

这两种状态在呈现上必须有明显区分（不同视觉样式），不能让用户混淆"事实"与"推测"。

---

## 4. 节点应包含的三层信息

每个节点（尤其是材料/工艺类节点）理论上应包含三层信息，三层的数据可得性差异很大（已调研确认，见第7节）：

1. **客观属性层**：不依赖应用场景的固有特性（物理/化学性质、工艺能力参数、成本区间）
2. **已知用法案例层**：当前已验证的实际应用场景，需可溯源
3. **行话/俗名映射层**：同一事物在不同行业、不同地区的别称（例如聚氯乙烯在不同语境下被称为PVC、亚克力、阳光板、树脂等）

---

## 5. 当前阶段目标：样板路径 Demo（不是完整产品）

**不做的事**：不铺开整张产业链图，不追求节点数量，不做共建机制，不做账号系统，不做变现功能。

**要做的事**：选择并做透**一条完整的示范路径**，从坐标系A走到坐标系B，做出一个具体的"啊哈"体验：

```
光伏 → 储能设备 → 电池隔膜（交汇点） → 聚乙烯/聚丙烯 → 塑料管道 / 包装薄膜（延伸应用）
```

这条路径前半段走坐标系A逻辑（行业上下游），后半段走坐标系B逻辑（材料属性延伸），中间在"电池隔膜"完成换轨。目标是做出一个能让人具体感受到"原来这个东西还连着完全没想到的领域"的最小可信样本，而不是追求覆盖面。

**用途**：自验证创意是否成立 + 作为报名/路演材料展示。

---

## 6. 已明确划出本体之外的分支方向（现阶段不做，但保留接口可能性）

以下内容经讨论后认为**不应该**进入图谱本体（会污染其客观性），但可能是未来基于图谱本体衍生的独立产品：

- **决策辅助类**：面向创业者的具体任务工具，例如"这条供应链的准入门槛在哪一层""谁是真正能对接的客户"（如"宁德时代不会直接收小厂货,但某大供应商可以"这类信息）
- **企业曝光/对接类**：设备/材料供应商在节点上付费曝光、被找到
- **专家协作/共建后台**：未来真正开放共建时，需要独立于展示界面的内容提交、审核系统
- **API/数据服务**：图谱本体未来可能作为基础设施，被其他产品调用

设计原则：现阶段的技术架构应让图谱本体和呈现层解耦，以便未来这些分支产品能直接复用本体数据，不需要推翻重做。

---

## 7. 数据来源现状（已调研，供数据生产阶段参考）

三层节点信息的可得性差异很大，与直觉相反：

- **客观属性层**：现有基础最好。材料科学领域已有成熟的标准化数据库（如材料基因工程数据库、国家材料数据标准化委员会、商业材料数据库），可作为起点借力，不需要从零造词汇表。
- **行话映射层**：有一定现成资源（化工类俗名学名对照表），但跨行业的俗称差异目前没有系统化整理，需要自行收集补充，且要注意同一俗名在不同语境下可能产生歧义。
- **已知用法案例层**：最薄弱的一环。市面上能找到的多是单一行业视角的产业链分析报告（"一个行业看自己的上下游"），几乎没有"一种材料/工艺横跨哪些不同行业被使用"的跨域案例数据库。这一层主要依赖人工收集、调研、从业者验证，目前没有可直接复用的现成数据源。

数据生产建议分工：人负责寻找资料、判断连接的真实性与可信度（不可外包）；AI coding agent 负责把已经过人工判断的资料转换为符合本体 Schema 的结构化数据（纯结构化转换工作，可放心交给AI）。

## 8. 技术选型（已定）

### 数据本体层
- 数据结构：图数据（节点 + 边），详见随附的 `schema.json`
- 存储：样板阶段使用结构化 JSON 文件；节点量增长到编辑体验明显变差时，升级为 SQLite（仍是单文件，无需独立数据库服务）；未来若需支撑多人共建、复杂图查询，再迁移到图数据库（如 Neo4j）

### 接口层
- 即使在样板阶段也通过轻量 API 获取数据，不让前端直接 import 本地 JSON 文件
- 技术选择：Node.js + Express，或 Next.js 内置 API routes
- 目的：为未来分支产品（共建后台、API开放）预留无痛升级路径

### 呈现/可视化层
- 前端框架：React，建议用 Next.js（利于未来 SSR/SEO）
- 图可视化引擎：**Cytoscape.js**（不是 react-force-graph 或纯手写 D3.js）
  - 原因：专为图谱场景设计，自带数据格式标准，性能和定制灵活度兼顾，适合长期演化、节点量持续增长、需要表达"已验证/待验证"视觉区分和"双坐标系换轨"交互的场景
- 样式：Tailwind CSS

### 部署
- 样板/早期阶段：Vercel
- 未来需要独立后端服务时，平滑过渡到 Serverless 函数或云服务器

### 数据生产辅助
- 使用 Claude Code 等 AI coding agent，将人工已核实的原始资料转换为符合 `schema.json` 的结构化数据，不用于判断连接真实性本身

---

## 9. 给 AI 编程协作工具的具体执行说明

1. 请严格按照随附的 `schema.json` 构建节点和边的数据结构，不要自行简化或更改字段含义，尤其是 `verification_status`、`coordinate_systems`、`evidence` 这几个字段是项目可信度的核心，不可省略。
2. 第一阶段开发范围仅限于第5节描述的样板路径，不要提前实现共建、账号、支付等第6节列出的分支功能。
3. 前端组件读取数据应通过 API 层，即便该 API 现阶段只是读取本地 JSON 文件并返回，也不要让组件直接 import 数据文件。
4. 图可视化部分使用 Cytoscape.js，节点应根据 `coordinate_systems` 和 `node_type` 呈现不同视觉样式，边应根据 `verification_status` 呈现不同线型/颜色（建议：verified 为实线，proposed 为虚线，并辅以颜色或图标区分）。
5. 随附的 `sample-data.json` 是已验证可用的样板数据，可直接作为开发和调试的初始数据集。
