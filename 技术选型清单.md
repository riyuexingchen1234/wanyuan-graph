# 万源图谱 —— 技术选型清单

## 前端框架
- **React**，建议用 **Next.js**（利于未来做服务端渲染、SEO，对希望长期被公众访问、被搜索引擎收录的产品是加分项）

## 图可视化引擎（三层混合架构）

万源图谱目标是百万到千万级节点，囊括几乎所有物理存在和概念存在。在此规模下，单一引擎无法兼顾"宏观震撼"与"微观精确阅读"，采用三层混合架构：

### 宏观层：3D 星云（Three.js）
- 用于产业全景入口，呈现"星云震撼感"，节点量级几千（产业级）
- 技术：**Three.js** + `Points`/`InstancedMesh` + `ShaderMaterial` + GPU picking（offscreen ID buffer）
- 依据：Three.js Points + ShaderMaterial 在桌面独显上百万级点实测 45-60fps；点云领域 Potree 已在浏览器跑十亿级点云（TU Wien 工业标准）；GPU picking 与规模无关，一帧完成
- 不用 react-force-graph 3D：开箱即用但定制天花板低
- 不用纯 D3 3D：开发量过大

### 中观层：WebGL 图谱（Sigma.js / Cytoscape WebGL）
- 用于展开某产业的子链/设备/工艺，节点量级万级
- 技术：**Sigma.js**（WebGL 原生，设计目标 10万+ 节点）优先；Cytoscape.js WebGL 预览作为备选
- 配合边捆绑（Pixel-Based Edge Bundling）减少视觉杂乱
- 依据：Sigma.js 被 Gephi Lite 采用，10万级是设计目标；Cytoscape.js WebGL 预览在 3200节点/68000边下仅 10fps（官方 blog），Canvas 渲染器更差（3fps），故 Cytoscape 仅作为微观层备选

### 微观层：2D 图谱（Cytoscape.js）
- 用于精确阅读单个实体的关系网，节点量级千级
- 技术：**Cytoscape.js**（Canvas，算法丰富、交互完善）
- 此规模下 Cytoscape 性能充足（1200节点/16000边 >100fps），且能完整呈现边的类型/可信度/标签
- 这是"精确阅读"层：verified 实线/proposed 虚线、关系类型切换、evidence tooltip

### 千万级演进路径（远期）
当节点量逼近千万级，纯前端一次性渲染不可行（物理限制），必须：
1. 服务端图数据库（Neo4j / Dgraph / Nebula）
2. 服务端 GPU 预布局（RAPIDS cuGraph 或 ngraph.native）——力导向布局在浏览器跑不动
3. 多分辨率八叉树预计算（仿 PotreeConverter）+ 前端视锥体流式加载
4. 二进制编码（protobuf，5M边/1M节点仅 23MB）
- 依据：Graphistry 实测 H100 跑 2B 边查询 2 秒；Potree 浏览器流式十亿级点云

### 不用的方案
- 不用纯 2D 一统到底：Cytoscape Canvas 在万级以上崩盘，无法承载百万级目标
- 不用纯 3D 一统到底：3D 下边的类型/可信度/标签可读性差，百万边交叉遮挡严重；精确阅读必须回到 2D
- 不用力导向全图布局：O(n²)~O(n³) 复杂度，大规模下用 Barnes-Hut O(n log n) 或服务端 FMM O(V+E)

## 数据存储
- **样板/早期阶段**：结构化 JSON 文件（节点+边，遵循 schema.json）
- **节点量增长、编辑体验明显变差时**：升级到 **SQLite**（仍是单文件，不需要独立数据库服务，可用 DB Browser for SQLite 之类工具方便编辑）
- **未来需要支撑多人共建、复杂图查询（如"查找所有横跨三个行业的材料"）时**：再迁移到图数据库，如 **Neo4j**
- 升级判断标准：不是看节点数量本身，是看"手动编辑数据是否已经痛苦到需要更好的工具"和"是否出现真实的多人写入/复杂查询需求"，这两个信号没出现之前不需要急着升级

## 接口层
- 即使在样板阶段，前端也通过一层轻量 API 获取数据，不直接 import 本地 JSON 文件
- 技术选择：**Node.js + Express**，或 **Next.js 内置 API routes**（如果前端用 Next.js，用这个更统一）
- 目的：让图谱本体和呈现层解耦，未来分支产品（共建后台、对外开放 API）可以无痛复用这一层，不需要推翻重做

## 部署
- **Vercel**（免费，Next.js 原生支持，一键部署，适合当前阶段）
- 未来如果需要独立后端服务（共建审核系统、开放 API 等），平滑过渡到 Serverless 函数（Vercel Functions / AWS Lambda）或云服务器

## 渲染/交互策略（性能与体验层面的设计原则，不是单纯的库选择）
- **不对全图做力导向布局**：力导向图算法的计算量随节点数量平方级增长，节点多了必然变卡，这是算法本身的数学性质，不是实现细节能优化掉的
- **采用渐进式局部展开**：默认只渲染用户当前关注节点 + 直接邻居，用户点击游走时才展开新的一圈关系，不一次性加载全貌，适用于物理流动链这种数量级较大的部分
- **材料属性延伸网保留全局视图的可能性**：材料种类数量相对有限可枚举，性能负担得起，适合做成更有"星图感"的独立小宇宙，作为特殊入口存在，不需要套用渐进式展开

## 数据生产辅助
- 使用 **AI coding agent**（如 Claude Code），按"多Agent协同岗位规则"中定义的分工，负责资料搜集、结构化转换；**判断信息真假的最终权力始终在人**，不交给 AI

## 暂不使用 / 明确避免的技术选项
- **不爬取明确禁止自动化采集的商业数据库**：Total Materia、ASM International 等，服务条款明确禁止
- **不依赖 AI 实时生成内容写入图谱本体**：已讨论否定，因为实时生成的内容因人因时而变，无法被稳定追溯和验证，违反"图谱必须可被反复查证"的根本要求
