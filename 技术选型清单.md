# 万源图谱 —— 技术选型清单

## 前端框架
- **React**，建议用 **Next.js**（利于未来做服务端渲染、SEO，对希望长期被公众访问、被搜索引擎收录的产品是加分项）

## 图可视化引擎
- **Cytoscape.js**
- 不用 react-force-graph：开箱即用但定制天花板低，长期会被迫迁移
- 不用纯手写 D3.js：灵活度最高但开发量太大，性能优化需要自己从零处理
- 选 Cytoscape.js 的原因：专为图谱场景设计，自带标准化的图数据格式，性能和定制灵活度兼顾，适合长期演化、节点量持续增长、需要表达 verified/proposed 视觉区分和"换轨"交互的场景

## 样式
- **Tailwind CSS**

## 数据存储
- **样板/早期阶段**：结构化 JSON 文件（节点+边，遵循 schema.json）
- **节点量增长、编辑体验明显变差时**：升级到 **SQLite**（仍是单文件，不需要独立数据库服务，可用 DB Browser for SQLite 之类工具方便编辑）
- **未来需要支撑多人共建、复杂图查询（如"查找所有横跨三个行业的材料"）时**：再迁移到图数据库，如 **Neo4j**
- 升级判断标准：不是看节点数量本身，是看"手动编辑数据是否已经痛苦到需要更好的工具"和"是否出现真实的多人写入/复杂查询需求"，这两个信号没出现之前不需要急着升级

## 接口层
- 即使在样板阶段，前端也通过一层轻量 API 获取数据，不直接 import 本地 JSON 文件
- 技术选择：**Node.js + Express**，或 **Next.js 内置 API routes**（如果前端用 Next.js，用这个更统一）
- 目的：让图谱本体和呈现层解耦，未来分支产品（共建后台、对外开放 API）可以无痛复用这一层，不需要推翻重做

## 部署
- **Vercel**（免费，Next.js 原生支持，一键部署，适合当前阶段）
- 未来如果需要独立后端服务（共建审核系统、开放 API 等），平滑过渡到 Serverless 函数（Vercel Functions / AWS Lambda）或云服务器

## 渲染/交互策略（性能与体验层面的设计原则，不是单纯的库选择）
- **不对全图做力导向布局**：力导向图算法的计算量随节点数量平方级增长，节点多了必然变卡，这是算法本身的数学性质，不是实现细节能优化掉的
- **采用渐进式局部展开**：默认只渲染用户当前关注节点 + 直接邻居，用户点击游走时才展开新的一圈关系，不一次性加载全貌，适用于物理流动链这种数量级较大的部分
- **材料属性延伸网保留全局视图的可能性**：材料种类数量相对有限可枚举，性能负担得起，适合做成更有"星图感"的独立小宇宙，作为特殊入口存在，不需要套用渐进式展开

## 数据生产辅助
- 使用 **AI coding agent**（如 Claude Code），按"多Agent协同岗位规则"中定义的分工，负责资料搜集、结构化转换；**判断信息真假的最终权力始终在人**，不交给 AI

## 暂不使用 / 明确避免的技术选项
- **不爬取明确禁止自动化采集的商业数据库**：Total Materia、ASM International 等，服务条款明确禁止
- **不依赖 AI 实时生成内容写入图谱本体**：已讨论否定，因为实时生成的内容因人因时而变，无法被稳定追溯和验证，违反"图谱必须可被反复查证"的根本要求
