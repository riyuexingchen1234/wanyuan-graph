## 1. 生产目标

我们要呈现的是一张**真实存在的网**：从原料到产品，从工艺到设备，从材料到需求，每一条连接都应该有依据、有出处、有可信度标注。

现实约束有两个：

- **理想数据难以结构化处理**。一份国家标准 PDF、一个上市公司年报里藏着上百条关系，靠纯人工抄录根本做不完。
- **人工处理成本不可承受**。一个产业数千节点、数十万条边，全靠专家人工录入，单产业成本可轻松突破百万，且每年还要更新。

A 路线的判断是：**机器为主、人工为辅、专家守门**。也就是 80% 的初始数据靠自动化产出，15% 靠社区共建补充和挑战，5% 关键边由专家终审并签发权威标记。这个比例不是出于"懒"，而是基于"自动化覆盖面广但会出错、众包能扩面但会失序、专家能定论但数量稀缺"的现实权衡——后面第 5 节会单独论证。

## 2. 六档可信度状态机

### 2.1 状态总览

| 状态 | 含义 | 谁可签发 | 谁可见 | 是否可作为权威引用 |
| --- | --- | --- | --- | --- |
| `auto-extracted` | AI 脚本从原始文本中抽取，未经任何人工核验 | 自动化管道 | 内部（不入用户视图） | 否 |
| `proposed` | 有依据但未验证，公开征求社区核验 | 自动化 + 众包提交 | 公开 | 否（必须配合状态标签使用） |
| `verified-community` | 社区/众包贡献者确认 | 社区（聚合规则） | 公开 | 可用于内部参考，不可作为单一权威 |
| `verified-expert` | 行业专家/资质审核员确认 | 审核员 | 公开 | **可作权威引用** |
| `disputed` | 被质疑，进入复核队列 | 任意用户/审核员 | 公开（高亮提示） | 否 |
| `deprecated` | 已废弃，附原因 | 审核员/自动化巡检 | 公开（弱化显示） | 否 |

> **设计推演**（待验证）：六个状态是否全部需要进入 schema 的 `verification_status` 枚举？从数据治理角度，`auto-extracted` 仅在 ETL 管道内流转更干净，呈现层只面对 5 档。**推荐做法**：`auto-extracted` 不入 schema，作为内部阶段标记；其余 5 档进入 schema v0.4 枚举。

### 2.2 状态转换规则

```
                 ┌────────────── deprecated（终态）
                 │      ↑
   auto-extracted ──→ proposed ──→ verified-community ──→ verified-expert
                       │   ↑             │   ↑                 │
                       │   │             │   │                 │
                       └─→ disputed ←────┘   └────── disputed ←┘
                            │
                            └──→ 复核后回到 proposed / verified-community / verified-expert / deprecated
```

**正向升级（晋升）**：

- `auto-extracted → proposed`：必须有 evidence 字段，evidence 中至少存在一条 `Source`。**不允许"裸三元组"进入 proposed**。
- `proposed → verified-community`：需要满足社区聚合规则，建议设置"N 个独立贡献者中至少 K 个确认"（K/N 阈值待定，可参考 Wikidata 的多源确认机制）。
- `proposed / verified-community → verified-expert`：由审核员手动签署，写入 `reviewed_by` / `reviewed_at`。

**横向挑战**：

- 任意状态 `→ disputed`：任何用户/审核员可发起质疑，需附 reason。**质疑不强制要求证据**，发现疑点即可——这是放低门槛、让长尾问题浮出来的关键设计。
- `disputed` 状态下，原 `verified-*` 等级**立刻失效**，相关边在呈现层高亮警示，停止作为权威引用。
- `disputed → *`：进入复核队列，由审核员判定回升或废弃。

**降级与废弃**：

- 任意状态 `→ deprecated`：源链接失效、专家主动更新、新证据根本性否定。废弃需附 `deprecated_reason`。
- `deprecated` 是终态，不直接回到其它状态。如需复活，应作为一条新的边重新创建（保留旧边用于审计追溯）。

### 2.3 举例

**例 1（正常晋升）**：AI 从《GB/T 23519-2009 聚苯乙烯树脂》标准文本中抽取出 `(聚苯乙烯, raw_material_for, 注塑成型)`，附上标准号 + 原文截图 → 状态 `auto-extracted`。运营核查 evidence 完整、术语边界清楚 → 升级为 `proposed`。三位社区贡献者分别确认 → 升级为 `verified-community`。审核员复核 → 升级为 `verified-expert`，写入 `reviewed_by = "expert-zhangsan"`, `reviewed_at = "2026-..."`。

**例 2（挑战与降级）**：某条 `verified-expert` 边 `(磷酸铁锂, applied_in, 储能电池)` 在 2027 年被一位社区贡献者质疑，理由"该专家当时引用的是 2022 年标准，2025 年新国标已调整应用范围" → 进入 `disputed`，原权威等级立即失效 → 审核员复核新国标，决定：(a) 边含义需修正 → 修订后回到 `proposed` 重新走流程；(b) 边已不成立 → 转入 `deprecated`，原边保留供审计。

**例 3（一票否决）**：审核员在月度抽查中发现 `verified-expert` 边 `(某材料, made_of, 某成分)` 的 evidence 链接已 404，且新公开的同主题国标中无此关系 → 审核员可单方将其降级为 `disputed`，无需经过社区环节。

## 3. 完整生产链路 7 步流程

每步用统一的元数据描述：**输入 / 输出 / 责任角色 / 工具 / 质量门**。

### 步骤 1：原始数据采集

| 维度 | 内容 |
| --- | --- |
| 输入 | 公开数据源 URL 清单、抓取计划 |
| 输出 | 原始文档（PDF/HTML/JSON），附带来源类型、获取时间、原文 hash |
| 责任角色 | 数据工程岗 |
| 工具 | cninfo 巨潮资讯爬虫（已有 `scripts/crawler/cninfo-crawler.ts`）、GB 标准爬虫（已有 `gb-standards-crawler.ts`）、stats.gov 国家统计局爬虫 |
| 质量门 | robots.txt 合规、URL 可重复访问、原文 hash 稳定 |

**数据源分级**（这是边界规则，不是建议）：

- **一级源（可作 verified 证据）**：国标全文公开系统、投入产出表、上市公司年报、行业协会团体标准、专利数据库（受 ToS 约束的除外）、官方统计。
- **二级源（仅可作交叉印证，不可单独支撑 verified）**：行业报告、券商研报、Wikipedia / Wikidata 等专业百科。
- **三级源（仅作辅助印证，**禁止**作为 verified 来源）**：新闻媒体、论坛、博客、自媒体。

> **设计推演**（待用户确认）：一级源是否需要二次核验（如多份国标交叉）？这取决于对"权威源可能犯错"的容忍度，本文暂按"一级源单一引用即可"，若严格可加"两份独立一级源"门。

### 步骤 2：AI 结构化抽取

| 维度 | 内容 |
| --- | --- |
| 输入 | 步骤 1 产出的原始文档 |
| 输出 | 三元组列表 `(head, relation_type, tail)` + 原始引用 + 推断置信度（0~1）+ 缺失字段标记 |
| 责任角色 | 自动化管道（LLM 抽取） + 数据运营复核 |
| 工具 | LLM（结构化抽取 prompt，限制输出 JSON schema）、自定义校验脚本 |

**强制输出字段**：
- `original_quote`：原文引用（精确字符串，含章节号/页码）
- `confidence`：模型自评置信度
- `missing_fields`：evidence 缺失时显式标记 `"evidence_incomplete"`

**拒绝策略（宁可丢错杀）**：
- 源文本自相矛盾 → 丢弃
- 抽取结果无原文支撑（仅靠"常识"）→ 丢弃
- 概念边界模糊（如"高科技"作为节点）→ 丢弃
- 关系类型不在 schema 枚举 → 进入"待人工归类队列"

**禁止行为**：
- AI **不得**自行将产出标记为 `verified-*`
- AI **不得**填写 `reviewed_by` / `reviewed_at`
- AI **不得**编造 `original_quote`

### 步骤 3：模式匹配与去重

| 维度 | 内容 |
| --- | --- |
| 输入 | 步骤 2 产出的三元组 + 已有节点库 |
| 输出 | 折叠后的节点（合并 `canonical_id`）+ 边 |
| 责任角色 | 自动化管道 + 运营抽查 |
| 工具 | Wikidata ID 匹配、Wikipedia 重定向、术语库（aliases） |

**关键场景**：
- **同物异名**："聚乙烯" ↔ "PE" ↔ "polyethylene" 通过 Wikidata Qcode 或 aliases 折叠。
- **跨产业节点合并**："银浆"在银浆产业和光伏产业各有一个节点（这是必然的——各产业有自己熟悉的命名），通过 `canonical_id` 折叠到同一规范节点，呈现层按需展开，编辑层保留各产业视角的独立节点（**这是 schema v0.3 已支持的设计**）。

**质量门**：canonical_id 折叠必须 **可回滚**——即每个产业视角的独立节点 id 永久保留，避免误折叠后无法拆分。

### 步骤 4：众包挑战层（proposed 状态的入口）

| 维度 | 内容 |
| --- | --- |
| 输入 | 任意用户/贡献者提交的三元组 + 依据 |
| 输出 | 新增的 `proposed` 状态边 |
| 责任角色 | 任意已注册用户 |
| 工具 | 提交表单、依据上传（链接/PDF/截图） |
| 质量门 | 提交即进入 proposed，**无门槛**，但必须填 `reasoning` 字段说明依据 |

**设计意图**：低门槛是为了让长尾事实浮出来。一条边先进入 `proposed`，它就进入了社区视野和审核视野；后续被验证则升，被证伪则降或废。**proposed ≠ verified**，下游使用方必须明确看到状态标签，不得将 `proposed` 边当作权威数据消费。

### 步骤 5：质疑与挑战（disputed 状态触发）

| 维度 | 内容 |
| --- | --- |
| 输入 | 任意用户对任意边发起质疑 + 理由 |
| 输出 | 该边进入 `disputed`，原 `verified-*` 等级立即失效 |
| 责任角色 | 任意用户/审核员 |
| 工具 | 质疑按钮 + 理由表单（不限字数） |
| 质量门 | 理由非空即可，**不要求证据** |

**为什么质疑不要求证据**：知识图谱最大的风险不是"被错怪"，而是"被沉默地错下去"。一个没有证据但言之成理的质疑，至少能将该边送入复核队列；一个需要证据才能质疑的设计，会让绝大多数普通用户放弃。

### 步骤 6：专家终审（verified-expert 产出环节）

| 维度 | 内容 |
| --- | --- |
| 输入 | 复核队列中的边（来自 `disputed` 或抽样审计） |
| 输出 | 终审结论：升级/维持/降级/废弃 + 审核记录（`reviewed_by`, `reviewed_at`） |
| 责任角色 | 审核员（具备可核实资质的专家） |
| 工具 | 审核后台（带边详情、evidence 列表、上下文） |
| 质量门 | 审核员身份可核实 + 全过程留痕 |

**审核员资质要求（建议）**：
- 可核实身份：学历证书、行业组织会员、职业经历中的可证明材料
- 利益声明：避免与被审核对象存在直接商业关联
- **多专家独立审核**机制：关键边至少 2 名审核员独立判断
- 审核员名单 **公开**，审核记录全留痕

**一票否决权**：审核员可单方将任何 `verified-*` 边降级为 `disputed`，无需经过社区环节。这是权威守门人的核心武器。

### 步骤 7：发布与持续监控

| 维度 | 内容 |
| --- | --- |
| 输入 | 状态变更事件流 |
| 输出 | 状态变更广播到所有消费方（前端 API、分支产品、订阅服务） |
| 责任角色 | 平台运营 |
| 工具 | 事件总线、订阅推送、定期审计脚本 |
| 质量门 | 每月随机抽样 N 条边，验证当前 evidence 是否仍站得住 |

**持续监控清单**：
- 源链接存活巡检（每周）
- 专家主动更新订阅（每季度）
- 用户报告处理（实时）
- 定期审计（每月抽样，年度全覆盖）

## 4. 数据来源边界（核心理念落地条款）

| 规则 | 解释 |
| --- | --- |
| AI 不得自行将 `proposed` 升级到 `verified-expert` | 防止 AI 幻觉污染权威层 |
| AI 不得填写 `reviewed_by` / `reviewed_at` | 这是"判断真假"的人类专属权 |
| 一级源可单独支撑 `verified` | 单一引用即可，前提是源本身可被独立验证 |
| 二级源仅可作交叉印证 | 不可单独支撑 `verified` |
| 三级源禁止作 `verified` 依据 | 媒体、论坛、博客的二手转述无法保证准确性 |
| 商业敏感数字免责 | 不对市场份额、具体价格数字承担权威责任；只到 `proposed` 层或更低 |
| `definition` 字段必须摘录自权威来源 | schema v0.3 已要求，AI 不得自行撰写定义 |

## 5. 80/15/5 比例的合理性论证

### 5.1 外部参照（已验证的通行做法）

> **标注**：以下两个项目我所知的基本事实是已确认的（OpenBase 是清华知识图谱开源项目，Wikidata 大量使用机器人编辑是公开的运营事实），但 **具体 80/15/5 数字** 来自社区资料汇编，**未在本文档写作时逐条核验最新统计**，请用户确认或替换为最新权威引用。

- **OpenBase（清华大学知识图谱开源项目）**：公开资料中描述其采用"自动化为主、社区众包补充、专家终审"的模式，与本流程设计同构。
- **Wikidata**：机器人编辑占比远高于人工编辑是公认事实；管理员仲裁（rollback、protect 等）仅占极小比例。Wikidata 的可持续运营本身就是"机器为主、人工为辅"思路的活证据。

### 5.2 为什么不能反过来（专家为主）

- **成本不可承受**：以电池产业为例，节点数千、潜在边数万。一个产业让 5 名专家审完需数月，扩展到全产业（覆盖材料/工艺/设备/产品/需求）需要专家数量超过任何现实可调度资源。
- **专家无法穷尽所有知识领域**：跨产业、长尾事实、跨语境歧义都不可能靠专家组解决。
- **专家本身也会出错**：单一专家的判断易带偏见，多专家成本又翻倍。

### 5.3 为什么不能全自动化

- **长尾事实**：标准未覆盖、公开报告未提及的连接大量存在。
- **跨语境歧义**：同一术语在不同产业含义不同，AI 缺乏领域语感。
- **定义边界判定**：节点 `definition` 的精确措辞决定下游所有应用，全靠 AI 撰写会引入系统性偏差。

### 5.4 80/15/5 的具体含义

- **80% 自动化**：覆盖 80% 的"显而易见型"边（标准中明示、年报中陈述、统计表中的流动关系）。
- **15% 众包**：覆盖 15% 的"长尾 / 跨语境"边（社区贡献者用经验补充、提出 `proposed`、挑战 `disputed`）。
- **5% 专家终审**：聚焦在 5% 的"高影响 / 高争议"边（关键产业的根基性关系、被质疑后需权威判断的边）。

## 6. 关键风险与防御机制

| 风险 | 攻击/失败场景 | 防御机制 |
| --- | --- | --- |
| 恶意贡献者大量提交伪 `verified-community` | 注册大量账号互相确认 | 审核员留痕 + 社区质疑 + 同 IP/设备指纹检测 |
| 共识陷阱：错误的"广泛共识"长期存在 | 错误边长期无人挑战，逐渐变成"常识" | 定期审计 + 专家一票否决 + 鼓励质疑的低门槛设计 |
| 专家俘获：被利益方收买 | 审核员被某公司/行业联盟定向影响 | 多专家独立审核 + 利益声明制度 + 审核员轮换 |
| 知识陈旧：标准更新但数据未更新 | 国标修订、年报改版后旧边仍在 | 源链接定期巡检 + 失效自动降级为 `disputed` |
| AI 幻觉污染 | LLM 编造看似合理的"边" | 强制 `original_quote` + 拒绝策略 + AI 不得签发 verified |
| 跨产业误折叠 | 看似同名实则不同物质的节点被合并 | canonical_id 折叠必须可回滚 + 独立 id 永久保留 |
| 商业敏感数字担责 | 引用过期价格、市场份额被引用方投诉 | 商业数字只到 `proposed` 层 + 免责声明 + 不写精确数值到 `attributes` |

---

## 7. 与万源核心理念的对齐说明

> 这一节是判断流程是否站得住的 **根本检验**：每条核心理念都必须在生产链路里找到对应落地条款，否则流程就是悬空的。

### 7.1 "客观呈现"

- 落地：所有边都附带 `evidence` 列表，每条 `evidence` 含 `source_type` / `description` / `url` / `retrieved_at`，依据链可点击。
- 反例检验：如果某条边没有 evidence 或 evidence 全是三级源，它就不应进入 `verified-*`。

### 7.2 "权威根基"

- 落地：`reviewed_by` / `reviewed_at` 字段（schema v0.3 已设计），审核员身份可核实、审核记录可追溯。
- 反例检验：如果 `verified-expert` 边的 `reviewed_by` 字段为空或指向 AI 标识，权威性立即崩塌。

### 7.3 "用户分类权"

- 落地：节点的 `aliases` 和 `attributes` 字段是用户视角的入口。用户视角的"对厨师有用的分组"通过节点 `attributes` 和 derived views（呈现层派生）实现，**不污染本体**——即不会为了某个用户群的需求去改 `node_type` 或强行加 `is_subtype_of` 边。
- 反例检验：若为了"对投资人更友好"而把节点 `parent_type` 改成"高景气产业"，就违反了"分类权归用户"原则。

### 7.4 "人是终点不是消费者"

- 落地：需求节点（`node_type = demand`，schema v0.3 已支持）作为图谱的一等公民，关系类型 `satisfies` 直接表达"X 满足 Y 需求"。
- 生产侧落地：`demand` 节点由运营人工编辑 + 社区提交产生（**AI 不主动生成 demand 节点**，因为需求的归纳本身带价值观判断）；边的关系类型 `satisfies` 也走完整的 6 档状态机。
- 反例检验：若把"消费者"作为超级节点连接所有需求，就是回到了"百川归海"的旧框架。

### 7.5 "服务还在张望的人"

- 落地：状态机公开透明、状态标签在 UI 中显著呈现——`proposed` 用虚线、`verified-expert` 用实线 + 审核员标签、`disputed` 用警示色。让"还在张望"的用户能看出每条边的可信度，而不是被"权威外观"误导。

## 8. 附录：与 schema v0.3 的兼容路径

由于 schema v0.3 的 `Edge.verification_status` 枚举只有 `verified` / `proposed` 两值，本文档提议的 6 档扩展需要：

1. **schema 升级到 v0.4**，新增 `auto_extracted` / `verified_community` / `verified_expert` / `disputed` / `deprecated` 五个枚举值。
2. **`Edge` 增加字段**：
   - `disputed_reason`（`string | null`）
   - `disputed_by`（`string | null`）
   - `deprecated_reason`（`string | null`）
   - `deprecated_at`（`date-time | null`）
   - `community_confirmations`（`array`，记录社区确认历史）
3. **数据迁移**：现有 `verified` 边默认映射到 `verified-expert`（前提是 `reviewed_by` 非空）；`proposed` 边保持不变。
4. **`auto-extracted` 不入库**：仅作为 ETL 管道内部状态，避免污染本体枚举。
