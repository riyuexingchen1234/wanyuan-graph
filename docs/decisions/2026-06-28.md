## 决策 #1：3 条跨产业边升级（不通过 verified-expert）

**执行细节**：
- `reviewed_by`: `community-principal-2026-06-28`

- `evidence`: 各 4 条（T/CPIA 0030 + 帝科/聚和/苏州固锝公开披露 + 华西证券研报）
- `note` 保留待复核事项：relation_type 边界、HJT 适用性

**推翻条件**：有真实行业专家接手 → 可在 reviewer_chain 追加 expert 角色 entry，action=approved，升级到 verified-expert。

## 决策 #2：`auto-extracted` 持久化

**议题**：6 档可信度中的 `auto-extracted` 是否作为持久化状态写入图谱数据。

**决策**：**持久化**。允许 `auto-extracted` 状态的边入库。

**依据**：
- 万源核心理念要求"可追溯"，80% 自动生产链的可见性是审计证据的一部分
- 临时态（仅 ETL 内）会让"哪些是脚本直接产出未经人眼"这一关键事实不可见
- 6 档枚举本身已包含该状态，schema v0.4 已就绪

**代价**：边数量增加、数据量上升，但这是透明度成本，可接受。

**推翻条件**：若后期发现 auto-extracted 边数量过大（>50%），可改用"仅在审核员审核后才入库"——但要保留内部 audit log。

## 决策 #3：AJV strict 模式

**议题**：schema-validator.ts 关闭 strict 模式（`strict: false, strictSchema: false`）。

**决策**：**保持关闭**，文件头注释说明取舍。

**依据**：
- schema.json v0.4 顶层带 `version`、`description` 等文档字段
- enum 类型带 `categoryDescriptions` / `statusDescriptions` 等说明字段
- AJV v8 默认 strict 模式会拒绝这些"未知 keyword"
- strict 模式是 linter，不是 validator——关闭不影响数据校验严格性

**代价**：未来若 AJV 升级到 v9/v10，需要重新评估严格性边界。

**推翻条件**：若团队坚持 strict 模式，可改用 `addKeyword` 显式注册说明字段为已知 metadata 字段。

## 决策 #5：众包身份门槛

**议题**：众包贡献者（提交 proposed 边 / 发起质疑）需要什么身份验证。

**决策**：**仅邮箱验证**。

**依据**：
- 真名 KYC 增加贡献门槛，与"用户分类权交还"理念冲突
- 当前阶段（v0.4 schema 刚落地、种子数据未上线）防刷不是首要问题
- 邮箱验证已可阻断最明显的批量机器人
- 中国互联网环境的真名认证有合规成本与用户抵触

**代价**：恶意贡献者可注册多邮箱刷边。

**缓解**：
- 质疑者 3 次被驳回 → 7 天禁言
- 提交边被多位审核员标记垃圾 → 邮箱黑名单
- v0.5 引入"贡献者信誉分"机制

**推翻条件**：出现规模化恶意贡献时再升级到手机号 / 实名。

## 决策 #6：质疑匿名性

**议题**：质疑者可否匿名？匿名到什么程度？

**决策**：**允许匿名质疑**，但有透明性下限：
- 边创建人可见：质疑者 ID / 邮箱 / 历史质疑次数 / 质疑类型
- 边创建人**不可见**：质疑者的真实姓名（即使已 KYC）/ 公司 / 行业
- 全图所有用户可见：质疑条数 + 质疑类型统计

**依据**：
- 全匿名会让恶意挑战者无所顾忌
- 全实名会形成寒蝉效应（尤其在涉及行业巨头/政治敏感产业时）
- "对边创建人透明 + 对全图公开计数" 是 GitHub issue 匿名举报的成熟模式

**代价**：边创建人收到匿名质疑时仍有心理压力。

**缓解**：质疑者必须填写具体疑点（≥30 字 + 质疑类型），纯情绪化质疑会被驳回。

## 决策 #7：临时禁言阈值

**议题**：质疑者被驳回几次后禁言。

**决策**：**3 次被驳回 → 7 天禁言**。

**依据**：
- 3 次是行业惯例（Twitter/X、StackOverflow、Reddit 都在此区间）
- 7 天避免永久误杀，给质疑者反思期
- 禁言期满后质疑记录清零，重新累计

**代价**：恶意质疑者被禁言后等 7 天换号继续。

**缓解**：邮箱黑名单 + IP 频率监控在 v0.5 引入。

## 决策 #8：利益冲突持股阈值

**议题**：审核员存在利益冲突（自身 / 亲属 / 所在公司持股）时如何判定。

**决策**：**5% 直接持股 + 直系亲属同行业任一条件触发回避**。

**依据**：
- 5% 是 A 股上市公司举牌线，是公司治理惯例阈值
- 直系亲属（父母 / 配偶 / 子女）的同行业就职构成强利益关联
- 间接持股、远亲不触发——避免过度泛化导致审核员不可用

**判定流程**：
- 审核员入驻时签署《利益声明书》
- 每条边审核前由系统弹出"是否回避"提示
- 故意隐瞒 → 取消审核员资格 + 已审边全部降级为 disputed

## 明确不拍（4 项）

以下 4 项**故意保持开放**，避免在数据生产期前过度设计：

### 不拍 #1：T/CPIA 子标准号是否必填
- **现状**：evidence 中已写"T/CPIA 0030.2-2021/2025"
- **不拍原因**：v0.4 阶段数据少，过度规范化拖慢速度
- **触发条件**：v0.5 引入数据审核自动化校验时再决定

### 不拍 #2：学术论文作为证据的接受度
- **现状**：v0.4 暂未明确，学术论文可作为 industry_report 来源
- **不拍原因**：学术论文存在预印本/已发表/被撤稿多状态，规则复杂
- **触发条件**：出现典型学术论文争议案例时再定

### 不拍 #3：申诉流程
- **现状**：审核员决策为终局，被驳回的质疑者无申述通道
- **不拍原因**：当前没有真实用户
- **触发条件**：v1.0 公测前必须有申诉通道，v0.5 期间设计

### 不拍 #4：状态机转换的边界条件细节
- **现状**：6 档状态机已定义转换规则（见 schema.json AllowedTransitions）
- **不拍原因**：边界案例（disputed 期间能否被再质疑？deprecated 能否复活？）需要真实数据驱动
- **触发条件**：首例 disputed / deprecated 案例出现时迭代
